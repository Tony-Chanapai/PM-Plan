================================================================
WONTECH ASIA · PM PLAN & LOGISTIC
SYSTEM CHANGELOG
================================================================

Two-app system for preventive maintenance tracking and device transport
logging at Wontech Asia. Built in-house from scratch, deployed on the
web, used daily by 9 engineers and admin.

URLs:
  PM Plan      pm-plan.netlify.app
  Logistic     wontech-logistic.netlify.app

Stack:
  Frontend     React 18 + Vite
  Backend      Supabase (PostgreSQL + REST + Storage)
  Hosting      Netlify (with serverless function for Maps URL parsing)
  AI           Anthropic Claude API (QC Report scan)


================================================================
v1.0  ·  INITIAL BUILD
================================================================

PM PLAN — preventive maintenance tracking
  - Device database (serial number, model, clinic, region)
  - Excel import for bulk device upload
  - 8 PM rounds per device (3M, 6M, 9M, 12M, 15M, 18M, 21M, 24M)
  - Engineer assignment per PM job (9 engineers)
  - QR code scanner (live video via getUserMedia + jsQR)
  - PM completion flow: scan → engineers → round → save
  - Light & dark theme
  - Admin user management
  - Activity log with CSV export

LOGISTIC — device transport tracking
  - Device search (serial number or QR code)
  - Transport logging with reason (Install / Demo / CS Service / Other)
  - Auto-fill district & province from Google Maps URL
  - Full transport history per device
  - Date-range PDF + CSV reports
  - Shared device pool with PM Plan (Install creates PM schedule)

SHARED SUPABASE BACKEND
  Tables: devices, pm_status, app_users, activity_logs, app_settings,
          logistics_devices, logistics_logs, user_favorites
  Storage: photos bucket (public)

v1.7  ·  ANDROID COMPATIBILITY
================================================================

PHOTO ATTACHMENT
  - Replaced hidden-input + button + inputRef.click() pattern (which
    fails on some Android browsers) with the standard <label>-wraps-
    <input> pattern that natively triggers file picker on every
    browser
  - Used "visually-hidden" CSS technique (1×1 clipped) instead of
    display:none — display:none breaks click handling on mobile Safari
    and older Android Chrome

GOOGLE MAPS URL PARSING (Netlify serverless function)
  - Multi-User-Agent retry strategy: tries desktop → android → ios →
    googlebot until one returns useful place info. Different UAs cause
    Google to return different redirect paths from the same short URL.
  - Smart place-name cleanup: handles Thai+English duplicated names
    (e.g., "Clinic Name คลินิก Clinic Name 1919...") by splitting on
    Thai chars and picking the cleanest English candidate
  - Direct Thai address fallback: parses "...Lat Yao, Chatuchak,
    Bangkok 10900" pattern when geocoding fails — extracts district
    and province from comma-separated parts
  - Place name extraction now searches: /place/Name/ URL path, ?q=
    query param, <meta og:title>, and <title> tag
  - Increased HTML body buffer 80KB → 200KB
  - Postcode-based query as additional geocoding strategy

================================================================
TECHNICAL DECISIONS
================================================================

DATA & SYNC
  - No device caching in db.js — always fetch fresh so Logistic
    location updates show immediately in PM Plan
  - Install on existing device PATCHES devices + DELETES pm_status
    to reset all 8 rounds (yellow warning shown to user first)
  - Install on new device INSERTS into both tables with start_date,
    end_date (+24mo), pm_dates (8 × 3-month intervals)

UI PATTERNS
  - Form inputs as memo'd components OUTSIDE parent with refs (fixes
    re-render typing lag)
  - PDF generation uses Blob URL + <a> click (no auto-print)
  - Light theme via [data-theme="light"] CSS vars + theme.js
    localStorage
  - Favorites in Supabase user_favorites table; Recents in
    localStorage (max 10 items)
  - Photo upload uses NO capture="environment" so user can pick
    from camera OR photo library

PERFORMANCE
  - Photos compressed client-side (1600px, 0.85 JPEG) before upload
  - PM status fetched in pages of 1000 rows (Supabase REST limit)
  - Dashboard fetches once on mount, filters client-side in useMemo

SECURITY
  - Row-level security on Supabase tables
  - Anon key safe to expose (public read for relevant tables only)
  - Admin role enforced via app_users.role column
